## Context

Today, `source: auto` resolution (`plan::resolve_source`) walks `mount_roots`, finds the first mounted volume whose contents match the profile's `ImportSource::detect`, and returns immediately — every other matching volume is silently dropped, with no trace in output or logs. `run_scan` and `run_import` (`src/lib.rs`) each call this once per invocation and drive a single scan → plan → (confirm →) execute → report sequence built around one resolved source root.

Explicit sourcing (`--source <path>` on the CLI, or a profile's `source: <path>` in YAML) is unaffected by any of this: it names exactly one path, always has, and this change leaves that path untouched.

The candidate directory name `resolve_source` already reads while probing (e.g. `/run/media/artiom/GOPRO075`) is the volume label — Linux automounters like udisks2 name mount points after the filesystem label — so "the name of the drive" the proposal asks for requires no new detection mechanism, only not discarding what's already read.

## Goals / Non-Goals

**Goals:**
- Every mounted volume matching a `source: auto` profile is scanned/imported in one invocation — none silently dropped.
- Each drive's name and path are shown before its plan/report, in both text and JSON output.
- A hard failure on one drive doesn't stop the batch — every detected drive still gets processed, and the run's exit code reflects the failure.
- `scan` and `import` share the same drive-enumeration and per-drive-header logic (no divergent detection between the two commands).

**Non-Goals:**
- No drive-name customization or aliasing in config — the displayed name is whatever the mount label already is.
- No concurrent/parallel drive processing — strictly sequential, one drive fully finishes (report printed) before the next starts.
- No new drive-metadata source (`lsblk`/`blkid`/udev) — name stays the existing mount-point basename.
- No change to explicit-source (`--source`, profile `source: <path>`) behavior, rendering, or JSON shape at all.
- No cross-drive deduplication — if the same footage somehow exists on two drives, both are processed independently.

## Decisions

**D1 — `resolve_source` splits by source mode, not by return arity alone.** Explicit sourcing (CLI `--source` or profile `source: <path>`) keeps today's single-path resolution untouched, feeding today's single-shot scan/plan/(confirm/)execute/report sequence with no loop and no JSON shape change. Only `SourceLocation::Auto` (and no `--source` override) goes through the new multi-drive path: a `resolve_sources` that walks every `mount_roots` entry, collects *every* directory where `detect()` returns true (instead of returning at the first), and yields them sorted by path. This keeps the blast radius scoped exactly to what the proposal calls BREAKING — explicit-source callers, tests, and scripts are provably unaffected because they never reach the new code path.
- *Alternative considered*: always return a `Vec` (length 1 for explicit sourcing) and always render the `drives`-array JSON shape, for one uniform code path. Rejected — it forces a JSON contract break on every caller, including ones with nothing to do with this feature, for a uniformity benefit that mostly matters internally. The proposal already commits to keeping explicit-source output byte-identical to today.

**D2 — Per-drive confirmation falls out of reusing the existing single-drive pipeline, not a new prompt mechanism.** `transfer::execute` already gates deletion behind `confirm()`/`assume_yes` once per call. The multi-drive loop simply calls the same scan → build plan → print plan → `transfer::execute` → print report sequence once per detected drive, in order. This automatically yields one prompt per drive with no new confirmation code, and keeps `--yes` behaving exactly as it does today (skips every prompt it would have skipped in a single-drive run).

**D3 — Failure handling is uniform across every per-drive step, and never stops the batch.** Any per-drive step that can fail today — building the plan (`plan::build_plan`/`build_scan_summary`), or `transfer::execute` — can still fail for one drive without changing what's possible for the next. A hard error (`Result::Err`) on drive *N* is caught right there in the loop, converted into that drive's recorded outcome, and the loop proceeds to drive *N+1*. This applies identically to `scan`, `import --dry-run`, and a real `import` — whichever per-drive step fails, that drive's outcome records the failure and the next drive still runs. Existing per-file soft failures (`TransferOutcome::Failed`) are already non-fatal to a single-drive run and need no new handling — they're just now also non-fatal to the *rest of the batch*.
- Each drive resolves to one of four outcomes: `completed` (plan/scan had entries, execution succeeded or wasn't attempted because `--dry-run`/`scan`), `completed_with_failures` (execution ran, at least one file/sidecar outcome was `Failed` — same check `run_import` already does today via `any_failed`, just scoped per drive), `error` (a hard `Err` from any per-drive step), or `empty` (the drive was detected but its plan/scan summary had zero entries — nothing to confirm or execute, distinct from an error).
- *Alternative considered*: stop the batch at the first failing drive. Rejected on reflection (see conversation) — drives are independent physical media; a bad file or a metadata-parse error on one card says nothing about the next, and stopping only strands already-inserted, already-detected drives behind a manual `--source <path>` workaround, which is the exact friction this feature removes. The cost of "wrong" here is a media file staying on a card one more day, not a corrupted destination — verification-before-delete (ADR 0003/0012) is unaffected either way.

**D4 — Multi-drive JSON is a `drives` array, keyed by nothing.** Each element: `{"name": string, "path": string, "status": "completed" | "completed_with_failures" | "empty" | "error", "error": string?, "summary" | "plan" | "results": <existing JSON type>?}` — the payload key and type match whatever the single-drive JSON already emits for that command (`scan` → `summary: ScanSummaryJson`; `import --dry-run` → `plan: PlanJson`; real `import` → `results: ResultsJson`), omitted (or `null`) for `empty`/`error` drives. The top-level document adds `"drives": [...]` and, for `import`, an aggregate `"any_failed": bool`; everything else about the document (`"profile"`, etc.) is unchanged. An array, not an object keyed by drive name, because volume labels aren't unique (a blank/reused card can read `NO NAME` on two drives at once) — the array preserves both without collision, and processing order already equals array order (D5).

**D5 — Drives are sorted by resolved path before processing.** `std::fs::read_dir` order is not guaranteed by the OS, so without an explicit sort, "drive 2 of 3" would be unstable between runs, making both the text output's numbering and any "which drive failed" debugging non-reproducible. Sorting by path (not by name, since names can collide) fixes this for text output, JSON array order, and the sequential processing order alike — one sort, three consumers.

**D6 — An `empty` drive (detected, zero groups) is reported, not skipped or conflated with "no sources".** Today `scan_profile` treats an empty plan the same as "no source resolved" — both print "no sources found". That collapse breaks under multi-drive: a drive that was found but had nothing to import is meaningfully different from a drive that was never inserted at all. `scan`/`import` now print that drive's header plus a "no media found" line for `empty` drives and move on — no confirmation prompt, no execute call, since there's nothing to act on.

**D7 — Exit code aggregates the same way it does today, widened across drives.** A run exits `Failure` if any drive's status is `error` or `completed_with_failures`; `Success` otherwise (including when every drive is `completed` or `empty`). `UsageOrConfig` is unchanged — it's still reserved for failures before the drive loop even starts (bad config, unknown profile).

## Risks / Trade-offs

- **[Risk] Duplicating the single-drive scan/plan/confirm/execute/report sequence between the explicit-source call site and the new auto-loop call site.** → Mitigation: factor that sequence into one function taking a resolved `(name, path)` pair, called once for explicit sourcing and once per detected drive for auto — D1's split is about output shape and JSON contract, not about forking the underlying logic.
- **[Risk] A hard error's message loses context once caught inside the loop instead of propagating to the top-level error handler.** → Mitigation: `Error`'s `Display` already includes the path involved in `Io`/`VerifyMismatch` variants (`src/error.rs`), so `error.to_string()` captured into the drive's `error` field carries the same detail the top-level handler would have printed; tracing at `-v`/`-vv` still logs per-drive progress as it does today.
- **[Risk] JSON consumers scripted against today's flat single-document shape for `source: auto` profiles break.** → Mitigation: explicitly scoped to `auto` (D1) — anything using explicit `--source` is provably unaffected; flagged BREAKING in the proposal so it's visible ahead of implementation, not discovered by a broken script later.
- **[Risk] Per-drive confirmation prompts could fatigue a user running many drives at once, training reflexive "yes" answers.** → Accepted trade-off, not mitigated: realistic drive counts here are 2-3 (USB hub of SD cards), each prompt sits directly under that drive's own freshly-printed plan (better-scoped context than one batched prompt covering N drives' worth of deletions at once), and `--yes` remains available for anyone who's decided in advance.
